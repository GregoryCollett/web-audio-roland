# TR-909 Web Audio Drum Machine — Design Spec

## Overview

A faithful recreation of the Roland TR-909 drum machine built with React and the Web Audio API. The UI is a modern dark reinterpretation of the 909 workflow — not a pixel-accurate hardware replica, but immediately recognizable in spirit. All sounds are synthesized from scratch using Web Audio oscillators, noise generators, and filters. No samples.

## Scope

### v1 (this spec)

- 16-step pattern sequencer
- All 11 TR-909 voices, fully synthesized
- Per-instrument parameters: level, tune (where applicable), decay
- Per-step accent
- Tempo control (BPM)
- Play/stop transport

### Future versions (out of scope for v1)

- Song mode (chaining patterns)
- Shuffle/swing
- Tap tempo
- A/B pattern variations
- Per-instrument tuning/decay/level knobs with full range
- Pattern copy/paste/clear

## Architecture

### Core Principle

The audio engine is fully external to React. React is a pure view layer — it reads state via `useSyncExternalStore` and dispatches actions by calling engine methods directly. React never creates or manipulates Web Audio nodes.

### Audio Engine

A single `AudioEngine` class owns all state and audio processing:

- `AudioContext` lifecycle
- Sequencer clock
- Transport state (playing, bpm, currentStep)
- Pattern data (steps, accents)
- Instrument parameters
- Voice triggering and the Web Audio node graph

#### Snapshot

The engine exposes state to React through an immutable snapshot:

```ts
type InstrumentId =
  | 'kick' | 'snare' | 'clap' | 'rimshot'
  | 'closedHat' | 'openHat'
  | 'lowTom' | 'midTom' | 'hiTom'
  | 'crash' | 'ride';

interface InstrumentParams {
  level: number;   // 0–1
  decay: number;   // 0–1, scaled to voice-appropriate range
  tune?: number;   // 0–1, only for voices that support tuning
}

interface EngineSnapshot {
  transport: {
    playing: boolean;
    bpm: number;
    currentStep: number; // 0–15
  };
  pattern: {
    steps: Record<InstrumentId, boolean[]>; // 11 instruments × 16 steps
    accents: boolean[]; // 16 steps
  };
  instruments: Record<InstrumentId, InstrumentParams>;
}
```

#### Subscription API

```ts
engine.subscribe(callback: () => void): () => void  // returns unsubscribe
engine.getSnapshot(): EngineSnapshot
```

A new snapshot object is created on every state change. The snapshot is small and cheap to create. During playback, snapshots update at the step rate (~8Hz at 120 BPM).

#### Mutation API

```ts
engine.init(): Promise<void>       // creates AudioContext (requires user gesture)
engine.play(): void
engine.stop(): void
engine.setBpm(bpm: number): void
engine.toggleStep(instrument: InstrumentId, step: number): void
engine.toggleAccent(step: number): void
engine.setParam(instrument: InstrumentId, param: string, value: number): void
```

### Sequencer Clock

Uses the standard Web Audio lookahead scheduling pattern:

1. A `setInterval` callback runs every ~25ms
2. It checks `AudioContext.currentTime` against the next scheduled note time
3. If a note is due within the lookahead window (~100ms), it schedules the note using `AudioContext.currentTime` for precise timing
4. Advances the step counter and notifies subscribers

This decouples audio timing from the main thread. Even if React is mid-render or the tab is throttled, scheduled notes play on time.

### Voice Architecture

Each instrument is a factory function:

```ts
type VoiceTrigger = (
  ctx: AudioContext,
  destination: AudioNode,
  time: number,
  params: InstrumentParams,
  accent: boolean
) => void;
```

When triggered, the function creates a short-lived Web Audio subgraph (oscillators, noise, filters, gain nodes), connects it to the destination, schedules parameter automation (envelopes), and lets the nodes be garbage collected after they finish.

No persistent oscillators or pre-allocated voice pools. Each trigger is independent.

#### Voice Synthesis Details

**Bass Drum:** Sine oscillator with pitch envelope sweeping from ~150Hz to ~50Hz. Short noise burst through a bandpass filter for the attack click. Amplitude envelope with fast attack, medium decay controlled by the decay parameter. Tune shifts the base frequency range.

**Snare:** Two layers — a tuned sine/triangle oscillator (~180Hz) for the body, plus bandpass-filtered white noise for the snappy top. Each layer has its own amplitude envelope. Decay controls the noise tail length. Tune shifts the body oscillator.

**Clap:** Bandpass-filtered white noise with a multi-trigger amplitude envelope — several rapid bursts (~4) followed by a longer decay tail. This creates the characteristic layered clap sound of the 909.

**Rimshot:** Two triangle oscillators at slightly detuned frequencies (~500Hz region) with very fast decay. Produces a metallic, punchy transient.

**Closed Hi-Hat:** Bank of ~6 square-wave oscillators at non-harmonic frequency ratios (e.g., 800, 1047, 1480, 1768, 2093, 2637 Hz) summed and run through a highpass filter. Very short amplitude envelope. Decay controls envelope length.

**Open Hi-Hat:** Same oscillator bank and filtering as closed hi-hat, but with a much longer decay envelope. Part of the hi-hat choke group.

**Hi-Hat Choke Group:** The engine holds a reference to the currently playing open hi-hat's gain node. When a closed hi-hat triggers, it immediately ramps that gain to zero, cutting off the open hat. This is the only case of persistent node reference tracking in the engine.

**Low Tom:** Sine oscillator with pitch envelope, targeting ~120Hz. Similar structure to kick but higher pitch range and shorter decay. Tune shifts the frequency.

**Mid Tom:** Same structure as low tom, targeting ~160Hz.

**Hi Tom:** Same structure as low/mid tom, targeting ~200Hz.

**Crash Cymbal:** Dense bank of metallic oscillators (more partials than hi-hat) through a bandpass filter with a long decay envelope. Produces a washy, sustained cymbal sound.

**Ride Cymbal:** Similar metallic oscillator bank to crash but with a narrower bandpass filter and moderate decay. More "bell-like" quality with a cleaner overtone structure.

## React Integration

### Hook Layer

```ts
// Module-level singleton
const engine = new AudioEngine();

// Primary hook — full snapshot
function useEngine(): EngineSnapshot {
  return useSyncExternalStore(engine.subscribe, engine.getSnapshot);
}

// Selector hooks — return stable references when slice hasn't changed
function useTransport(): EngineSnapshot['transport'] { ... }
function usePattern(): EngineSnapshot['pattern'] { ... }
function useInstrumentParams(id: InstrumentId): InstrumentParams { ... }
```

Selector hooks use `useSyncExternalStore` with selector-specific `getSnapshot` functions that return the same reference when the relevant slice hasn't changed. This prevents unnecessary re-renders — e.g., a knob component that only cares about one instrument's params won't re-render on step advancement.

### Action Dispatch

Components import the engine singleton directly and call methods:

```tsx
<button onClick={() => engine.toggleStep('kick', 3)} />
<button onClick={() => engine.play()} />
```

No context providers or prop drilling for actions.

### AudioContext Initialization

The Web Audio API requires a user gesture to create an `AudioContext`. The app shows a one-time "click to start" overlay. Clicking calls `engine.init()` which creates the `AudioContext` and resolves. Before init, `getSnapshot()` returns a default state with `playing: false`, `currentStep: 0`, and default patterns/params.

## UI Design

### Visual Direction

Modern dark theme with the 909's orange (#ff6b35) as the accent color. Monospace typography. Clean, minimal chrome. Controls designed for mouse interaction, not mimicking hardware knobs.

### Layout (top to bottom)

1. **Transport bar** — "TR-909" title + "RHYTHM COMPOSER" subtitle, BPM display (editable), Play/Stop buttons
2. **Instrument selector** — horizontal row of 11 voice tabs (BD, SD, CP, RS, CH, OH, LT, MT, HT, CC, RC). Selected instrument is highlighted orange.
3. **Parameter knobs** — rotary knob controls for the selected instrument's level, tune (if applicable), and decay. Displayed in a recessed dark panel.
4. **Step grid** — 16 step buttons grouped in 4s (matching beat groupings). Active steps are lit orange. Steps are toggled on click.
5. **Accent row** — thin toggle row below the step grid for per-step accent on/off.
6. **Playhead** — thin indicator bar below the accent row showing the current step during playback.

### Component Tree

```
App
├── InitOverlay          — "Click to start" gate for AudioContext
├── Transport            — play/stop/bpm
├── InstrumentSelector   — 11 voice tabs
├── ParamKnobs           — rotary controls for selected voice
├── StepGrid             — 16 step buttons for selected instrument
├── AccentRow            — 16 accent toggles
└── Playhead             — current step indicator
```

### Interaction Details

- **Step buttons:** Click to toggle. Active steps show solid orange fill. Inactive steps show dark background.
- **Instrument tabs:** Click to select. Shows that instrument's pattern in the step grid and its params in the knobs.
- **Knobs:** Click-and-drag vertically to adjust. Visual feedback via a conic gradient arc.
- **BPM:** Click to edit, type a value. Reasonable range: 40–300.
- **Playhead:** Purely visual — a thin orange bar under the current step, advances in real time during playback.

## Testing

### Engine Tests (Vitest)

- Pattern toggling: `toggleStep` flips the correct cell, new snapshot returned
- Transport: play/stop state transitions, BPM changes, step advancement
- Snapshot immutability: each mutation produces a new object reference
- Hi-hat choke group: closed hat silences open hat
- Clock scheduling: notes scheduled at correct `AudioContext.currentTime` offsets for a given BPM

Uses `OfflineAudioContext` where possible, mock `AudioContext` for timing-sensitive tests.

### Component Tests (Vitest + React Testing Library)

- Step buttons toggle on click
- Instrument selector switches the displayed pattern and params
- Transport buttons call engine.play/stop
- Playhead position reflects currentStep from snapshot

### Manual Testing

Sound design is inherently audible. Automated tests verify node creation and connections; tuning the synthesis to sound like a real 909 requires listening and iterating on parameter values.
